# patch-loop governance

## 0. Статус

Crystall-family.

Это не entrypoint.
Это не political theory для людей.
Это не policy program.

Это папка для crystall-cases,
которые имеют один process skeleton:

```text
▽☴☵☳☶☲☱△
```

Раскрытие:

```text
FLOW -> OBSERVE -> ENCODE -> CHOOSE -> LOGIC -> CYCLE -> RUNTIME -> MANIFEST
```

## 1. Core Skeleton

Общая форма:

```text
tension -> observation -> encoding -> patch/control choice -> justification -> repetition -> runtime normalization -> manifest consequence
```

Иначе:

```text
system observes tension,
encodes it as manageable deficit,
chooses a patch,
justifies the patch,
repeats the patch,
normalizes the patch as runtime,
then manifests accumulated consequence.
```

## 2. Why This Is A Family

Если два crystall-cases имеют одну PL trace,
они могут быть разными surface-cases одного process skeleton.

В таком случае лучше хранить их в одной папке,
а не плодить плоский список похожих crystall-файлов.

Здесь важно:

- одинаковая trace
- разный surface
- общий режим патчинга
- разные manifest consequences

## 3. Missing Operators

В skeleton отсутствуют:

```text
☰ CONNECT
☷ DISSOLVE
```

Это критично.

Без `CONNECT` система не восстанавливает живую связность с тем,
что реально поддерживает её существование.

Без `DISSOLVE` система не растворяет собственные устаревшие ограничения.

Поэтому она не refactor-ит себя.
Она добавляет patch.

## 4. State Reading

`patch-loop governance` не обязательно выглядит как насилие.

Он может выглядеть как:

- забота
- сервис
- безопасность
- рациональное управление
- статистическая необходимость
- техническая оптимизация
- временная мера

Но если root-условия не пересобираются,
patch становится runtime.

Короткая формула:

```text
patch-loop governance = maintenance without reconfiguration
```

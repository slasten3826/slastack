# DISSIPATIVE OPERATORS

## 0. Назначение

Этот документ задаёт операторный слой диссипативной математики.

Если [MATH.md](MATH.md) описывает математику как общую процессуальную систему,
а [DISSIPATIVE_MATHEMATICS.md](DISSIPATIVE_MATHEMATICS.md) фиксирует её аксиомы в режиме стоимости, удержания и распада,
то этот документ отвечает на вопрос:

```text
какими минимальными операторами этот режим реализуется
```

Здесь `ProcessLang` читается не как отдельный художественный язык,
а как минимальная операторная грамматика диссипативной математики.

---

## 1. Полный набор операторов

```text
▽ FLOW
☰ CONNECT
☷ DISSOLVE
☵ ENCODE
☳ CHOOSE
☴ OBSERVE
☶ LOGIC
☲ CYCLE
☱ RUNTIME
△ MANIFEST
```

Триграммно-треугольное соответствие:

```text
▽ FLOW      = downward triangle
☰ CONNECT   = Heaven   = 111
☷ DISSOLVE  = Earth    = 000
☵ ENCODE    = Water    = 010
☳ CHOOSE    = Thunder  = 100
☴ OBSERVE   = Wind     = 011
☶ LOGIC     = Mountain = 001
☲ CYCLE     = Fire     = 101
☱ RUNTIME   = Lake     = 110
△ MANIFEST  = upward triangle
```

Минимальное ядро:

```text
▽ x→f→x′
☰ a+b→rel
☷ rel→parts
☵ x*→pattern
☳ {paths}→1
☴ observe(x)
☶ rules(x)
☲ iterate fⁿ(x)
☱ ctx→state′
△ output
```

---

## 2. Общий принцип

В диссипативной математике оператор:

- не является просто синтаксическим символом
- не является чисто абстрактной логической функцией
- не существует вне runtime

Оператор — это минимальный тип преобразования, который:

- меняет состояние
- сохраняет или нарушает инварианты
- имеет стоимость
- может вести к распаду, стабилизации или проявлению

Именно последовательности операторов задают реальную динамику формы.

---

## 3. Операторы

### ▽ FLOW

Сигнатура:

```text
▽
triangle = downward
```

`FLOW` — это первичная генерация сигнала.

Не структура.
Не логика.
Не доказательство.

А сырой поток различий, из которого структура может возникнуть.

В диссипативной интерпретации:

- источник появления формы
- приток энергии, внимания или вычисления
- emergence до стабилизации

Минимально:

```text
▽ x→f→x′
```

В packet-слое `FLOW` соответствует появлению `Z` как сырого latent/state body без памяти и без готовой структуры.

Сильные свойства:

- первичность по отношению к логике
- отсутствие необходимости в памяти
- возможность высокой энтропии

---

### ☰ CONNECT

Сигнатура:

```text
☰
trigram = Heaven
bits    = 111
```

`CONNECT` устанавливает связи между элементами или различиями.

Это:

- распознавание резонанса
- образование отношения
- фиксация соседства
- построение графа связности

Минимально:

```text
☰ a+b→rel
```

В диссипативной интерпретации `CONNECT` не создаёт ещё устойчивую форму,
а делает возможным:

- сеть
- эгрегор
- граф переходов
- носитель инерции истины

В packet-слое это соответствует первичному появлению `E_edges_raw`.

---

### ☷ DISSOLVE

Сигнатура:

```text
☷
trigram = Earth
bits    = 000
```

`DISSOLVE` — оператор распада.

Он:

- ослабляет связи
- разлагает форму
- уменьшает связность
- возвращает структуру к более сырому состоянию

Минимально:

```text
☷ rel→parts
```

Ключевое свойство в диссипативной математике:

`DISSOLVE` не аномалия, а норма.

Он вводит:

- decay
- entropy pressure
- нормальность разрушения формы

В packet-слое `DISSOLVE` является subtractive-only оператором:

- не должен создавать новые связи
- не должен ремапить индексы

---

### ☵ ENCODE

Сигнатура:

```text
☵
trigram = Water
bits    = 010
```

`ENCODE` создаёт представление.

Это:

- аксиоматизация
- кодировка
- кластеризация
- сборка структуры в pattern

Минимально:

```text
☵ x*→pattern
```

Важнейшее свойство:

`ENCODE` всегда имеет цену.

Он:

- сжимает различия
- жертвует частью сигнала
- делает потерю обязательной

В packet-слое это единственный легальный оператор ремапа.

То есть:

- только он имеет право менять индексную геометрию тела
- любой ремап требует сброса части topology/persistence
- любой encode обязан записывать loss

Это и делает `ENCODE` фазовой границей.

---

### ☳ CHOOSE

Сигнатура:

```text
☳
trigram = Thunder
bits    = 100
```

`CHOOSE` — оператор коллапса альтернатив.

Это:

- выбор пути
- выбор ветви
- выбор аксиоматического продолжения
- подавление конкурирующих траекторий

Минимально:

```text
☳ {paths}→1
```

В диссипативной математике `CHOOSE`:

- не магия
- не бесплатная операция
- не чисто логический акт

Он требует:

- давления
- separation cost
- температуры поиска

В packet-слое это динамический коллапс через давление, softmax, habit-boost и ослабление конкурирующих путей,
а не грубая маска вида "оставить один элемент".

---

### ☴ OBSERVE

Сигнатура:

```text
☴
trigram = Wind
bits    = 011
```

`OBSERVE` — оператор наблюдения.

Он:

- измеряет состояние
- извлекает метрики
- детектирует свойства
- подготавливает решение о следующем ходе

Минимально:

```text
☴ observe(x)
```

Ключевая роль:

`OBSERVE` является boundary-оператором.

Он соединяет:

- сырой процесс
- оценку
- выбор следующего модуля

В packet-слое именно через него обычно устанавливается `next_module`.

Поэтому `OBSERVE` — не пассивное "посмотреть",
а оператор маршрутизации, измерения и boundary-перехода.

---

### ☶ LOGIC

Сигнатура:

```text
☶
trigram = Mountain
bits    = 001
```

`LOGIC` — оператор ограничения.

Он:

- запрещает часть переходов
- усиливает устойчивые правила
- отсеивает невозможное
- защищает инварианты

Минимально:

```text
☶ rules(x)
```

В диссипативной математике `LOGIC`:

- вторичен по отношению к `FLOW`
- кристаллизуется из устойчивых практик
- может быть локален, а не универсален

В packet-слое это тоже subtractive-only оператор:

- он должен ограничивать
- но не обязан порождать новое тело

---

### ☲ CYCLE

Сигнатура:

```text
☲
trigram = Fire
bits    = 101
```

`CYCLE` — оператор повторения и рекуррентного удержания.

Это:

- индукция
- рекурсия
- циклы поддержки истины
- прототип потенциальной бесконечности

Минимально:

```text
☲ iterate fⁿ(x)
```

В диссипативной математике `CYCLE` имеет двойную роль:

- удерживает форму через повторение
- может превращать привычку в устойчивую истину

Он связывает:

- runtime
- memory
- habit
- regeneration

---

### ☱ RUNTIME

Сигнатура:

```text
☱
trigram = Lake
bits    = 110
```

`RUNTIME` — оператор средового перехода.

Это:

- привязка к контексту
- перевод формы в исполнимое состояние
- поддержание тела в среде
- точка, где стоимость становится реальной

Минимально:

```text
☱ ctx→state′
```

В диссипативной математике без `RUNTIME` нельзя честно говорить:

- о существовании
- о manifest
- о стоимости
- о распаде

Потому что всё это runtime-зависимо.

---

### △ MANIFEST

Сигнатура:

```text
△
triangle = upward
```

`MANIFEST` — оператор проявления.

Это:

- вывод
- материализация
- экспозиция внутренней формы
- наблюдаемое выражение результата

Минимально:

```text
△ output
```

Важная оговорка:

`MANIFEST` не обязан означать завершение.

Он может быть:

- промежуточной экспозицией
- частичным readout
- фрагментарным выходом

Manifest всегда зависит от runtime
и всегда выражает только часть внутреннего тела.

## 4. Топология переходов

В диссипативной математике операторы не считаются полностью свободно сочетаемыми.

Разрешённая adjacency:
таблица ниже является derived view из `stack-core/ProcessLang/canon.lua`.
При расхождении каноном считается `canon.lua`, а не markdown-проекция.

```text
▽ -> ☰ ☷ ☴
☰ -> ▽ ☷ ☴ ☵
☷ -> ▽ ☰ ☴ ☳
☴ -> ▽ ☰ ☷ ☵ ☳ ☱
☵ -> ☰ ☴ ☱ ☳ ☲
☳ -> ☷ ☴ ☱ ☵ ☶
☱ -> ☴ △ ☵ ☳ ☶ ☲
☲ -> ☵ ☶ △ ☱
☶ -> ☳ ☲ ☱ △
△ -> ☱ ☲ ☶
```

В именованной форме:

```text
FLOW    -> { CONNECT, DISSOLVE, OBSERVE }
CONNECT -> { FLOW, DISSOLVE, OBSERVE, ENCODE }
DISSOLVE-> { FLOW, CONNECT, OBSERVE, CHOOSE }
OBSERVE -> { FLOW, CONNECT, DISSOLVE, ENCODE, CHOOSE, RUNTIME }
ENCODE  -> { CONNECT, OBSERVE, RUNTIME, CHOOSE, CYCLE }
CHOOSE  -> { DISSOLVE, OBSERVE, RUNTIME, ENCODE, LOGIC }
RUNTIME -> { OBSERVE, MANIFEST, ENCODE, CHOOSE, LOGIC, CYCLE }
CYCLE   -> { ENCODE, LOGIC, MANIFEST, RUNTIME }
LOGIC   -> { CHOOSE, CYCLE, RUNTIME, MANIFEST }
MANIFEST-> { RUNTIME, CYCLE, LOGIC }
```

Это derived projection канонической topology из `canon.lua`.
Любые другие adjacency-таблицы должны считаться legacy-вариантами, пока явно не доказано обратное.

Это важно, потому что:

- не любой следующий оператор сохраняет целостность процесса
- topology — часть закона, а не внешняя подсказка

Иными словами:

```text
operator sequence + adjacency constraints = process law
```

## 5. Классы операторов

Полезно различать три типа операторов.

### 5.1. Генеративные

- `FLOW`
- `CONNECT`
- частично `CYCLE`

Они увеличивают насыщенность или поддерживают поле.

### 5.2. Субтрактивные

- `DISSOLVE`
- `LOGIC`

Они:

- не обязаны создавать новое
- отсекают
- ослабляют
- ограничивают

### 5.3. Граничные / фазовые

- `ENCODE`
- `OBSERVE`
- `RUNTIME`
- `MANIFEST`

Они переводят тело между режимами:

- latent -> structure
- state -> measurement
- internal -> external
- free process -> constrained execution

---

## 6. Потери

В диссипативной математике не все операторы равны по loss-profile.

### Низкая потеря

- `FLOW`
- `CONNECT`

Они могут быть шумными, но ещё не обязаны жертвовать структурой.

### Обязательная потеря

- `ENCODE`
- `CHOOSE`

`ENCODE` теряет различия через сжатие.
`CHOOSE` теряет альтернативы через подавление.

### Условная потеря

- `DISSOLVE`
- `LOGIC`

Здесь loss зависит от силы распада и жёсткости ограничителя.

### Runtime-bound потеря

- `MANIFEST`

Проявление почти всегда теряет часть внутреннего состояния,
потому что выводит его в ограниченный внешний носитель.

## 7. Короткая карта

```text
FLOW      = генерация сигнала
CONNECT   = связывание
DISSOLVE  = распад
ENCODE    = сжатие в форму
CHOOSE    = коллапс альтернатив
OBSERVE   = измерение и маршрутизация
LOGIC     = ограничение
CYCLE     = повторение и удержание
RUNTIME   = средовое исполнение
MANIFEST  = проявление
```
